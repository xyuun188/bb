# 苍鸮量化策略闭环治理计划（2026-06-20）

## 总目标

- 恢复“发现机会 → 评估收益 → 合理开仓 → 可解释持仓/平仓 → 复盘学习”的完整闭环。
- 清理历史遗留的隐藏固定阈值，所有会影响开仓、仓位、平仓、训练的参数必须进入可审计策略快照。
- 消灭误导性状态：不能再出现“已进入执行队列但实际未下单”“弱证据/小仓原因不清”“OKX 拦截后看不到规则细节”。

## 当前批次范围

1. 仓位 sizing：强质量信号不能被旧探针上限长期压成小仓；弱证据仍必须小仓或不下单。
2. 亏损复开：同币种同方向亏损后，短时间重复开仓必须有更强新证据。
3. 快平仓：几分钟内亏损平仓必须由强风控证据触发，不能因普通噪声频繁止损。
4. OKX 执行：下单前读取交易规则；若交易所仍拒绝，记录结构化原因、规则、计划数量和最终请求。
5. 训练数据：污染样本降权/排除，训练面板显示数据来源、动作分布、新鲜度和质量。
6. 模型监控：平台只允许 18000/18001/18002 合约，禁止 8003；DeepSeek、Qwen、本地量化工具状态必须一致展示。

## 验收标准

- `release_pressure_active` 只由真实满仓、碎片化和低质量仓压力触发；历史亏损不能全局压死新仓。
- `expected_net_return_pct` 拆出 AI、ML、时序、情绪、历史、手续费、滑点和风险惩罚来源。
- 每个新仓都能解释：为什么开、为什么这个仓位、为什么这个杠杆、OKX 最小规则如何满足。
- 每个不开仓都能解释：是哪一项证据不足或风险项超限，而不是泛泛“动态证据不足”。
- 本地测试、安全扫描、线上冒烟、Git、Hindsight 更新后，才允许标记为完成。

## 线上观察复查清单

### 30 分钟技术复查

- 确认市场分析和持仓分析都有新增记录，不能只跑持仓或只跑市场。
- 确认 `decision_interval_seconds` 与后台配置一致，当前目标为 30 秒。
- 确认服务器监控中 Qwen、DeepSeek、本地量化工具和平台服务器均有数据，且只使用 18000/18001/18002 隧道。
- 确认系统自检 `critical=0`、`warning=0`，若出现异常必须记录具体检查项。
- 确认最近日志没有 `maximum context length`、`Invalid local AI tools API key`、OKX 最小下单数量拦截。

### 2-4 小时策略复查

- 统计市场候选数、开仓信号数、真正提交订单数、交易所拒单数，判断是否仍被执行链路卡死。
- 抽查最近 20 条市场分析，确认 `expected_net_return_pct` 不再固定集中在约 `-0.46%`。
- 抽查所有正预期候选，确认不开仓原因能具体落到收益、亏损概率、尾部风险、模型缺失、行情质量或 OKX 规则。
- 检查强质量信号是否被 `probe_cap` 压成极小仓；若被压制，必须查看 `profit_risk_sizing.strategy_quality_override` 和 `quality_tier`。
- 检查同币种同方向亏损后短时间重复开仓是否显著减少；若重复开仓，必须有更强新证据字段。
- 检查几分钟内亏损平仓是否仍频繁出现；若出现，必须看到强风控触发原因。

### 24-72 小时盈利闭环复查

- 按已平仓订单统计手续费后净收益、平均盈利、平均亏损、盈亏比和最大单笔亏损。
- 对比开仓仓位分布，确认仓位大小能由预期净收益、亏损概率、尾部风险、账户余额和 OKX 规则解释。
- 检查本地 ML、服务器盈利模型、时序模型、情绪模型是否持续返回有效贡献，而不是长期 0 或未返回。
- 检查训练数据质量报告，确认异常行情、手动/测试单、快亏损平仓样本没有进入主训练样本。
- 若订单不足 20 笔，以样本数优先，不用时间强行判断盈利能力。

### 复查时必须读取的页面/API

- 主面板运行状态：确认市场/持仓两条线分开运行。
- 分析记录详情：查看 `expected_net_breakdown`、`profit_risk_sizing`、`execution_steps`。
- 开仓漏斗诊断：统计候选进入、收益过滤、风控复核、OKX 校验、订单提交各阶段数量。
- 服务器监控：确认模型端点、平台服务器、自检结果一致。
- 数据采集管理：确认 Scrapling/zvec 数据源状态和训练数据新鲜度。

### 若仍不开仓的定位顺序

1. 先看开仓漏斗是否卡在执行层、收益层、风控层、行情质量层或交易所校验层。
2. 再看最近正预期候选是否存在，并读取 `expected_net_breakdown` 的最大负贡献项。
3. 再看 `profit_risk_sizing` 是否显示 `quality_tier`、`strategy_quality_override`、`probe_cap_applied`。
4. 再看 OKX 规则校验是否给出最小张数、最小名义价值、步进精度和最终下单数量。
5. 最后看模型健康，确认本地 ML 和服务器盈利模型是否可用，不能用泛泛“策略保守”解释。
